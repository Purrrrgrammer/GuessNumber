Применила State Pattern для управления состоянием игры. 
Использовала фабрику состояний (IGameStateFactory) для устранения жесткой связи между состояниями и соблюдения DIP. 
Каждое состояние получает зависимости через конструктор, что обеспечивает SRP и тестируемость.


SRP 
Каждый класс имеет только одну зону ответственности
GameService - управляет процессом игры
GameStateFactory - создает состояния игры
каждый GameState - обрабатывает соответсвующее состояние игры
OCP
реализован паттерн "Состояние". При добавление нового состояния, нужно создать новый класс и метод в фабрике. 
Старый код не трогаем
LSP
Все состояния взаимозаменяемы через IGameState
ISP
интерфейсы "узкие". IGameState = 1 метод, IGameStateFactory = 4 метода
DIP
Все зависимости через интерфейсы, в консольном приложении компоновка в Program. 
В приложении Web API можно использовать DI-контейнер
